# Plane Calculator v1.0 Scope

## 1. Purpose

本文档用于从 `PlaneDesign.md` 的完整功能池中，切出 Plane Calculator v1.0 的可落地范围，并作为 v1.0 范围冻结的基准文档。

它的作用是：

- 明确哪些能力属于 v1.0 主范围，哪些只做预留，哪些后置，哪些明确不做。
- 为后续范围冻结、任务拆分和实现排期提供统一依据。
- 避免把 `PlaneDesign.md` 中的长期设计，直接当作第一版默认目标。
- 让后续 `Plane_v1.0_CodexTasks.md` 的拆分有清晰、稳定的边界。

本文档不是：

- Plane Calculator 的完整产品设计文档
- 新一轮架构审计文档
- Codex 直接实施任务清单
- 当前代码现状的“已实现清单”

后续可以基于本文档继续拆出 `Plane_v1.0_CodexTasks.md`，但那属于下一阶段的实施拆分，不属于本文件范围。

## 2. Product Positioning

Plane v1.0 的定位不是“把所有二维数学能力一次性做满”，也不是把 Plane 做成默认就很复杂的专业创作软件。

它应被定义为：

> Plane v1.0 is a clean 2D mathematical art canvas with a traditional graphing/geometry entry experience and expandable advanced creation capabilities.

中文表达可以概括为：

- 默认界面克制，先给用户一个干净、熟悉的二维数学画布。
- 中央主体是标准二维坐标画布，而不是一开始就铺满面板。
- 入口体验接近传统几何画板 / 函数画板，保留熟悉的 graphing / geometry entry experience。
- 底层方向仍然服务于 eMathica 的“数学艺术软件”定位。
- 高级创作能力应可扩展，但不应在默认打开时全部暴露。

## 3. v1.0 Design Principles

- 默认界面简单，不展示过多内容，不制造“复杂专业设计软件”的压迫感。
- 画布优先，面板按需展开，默认先让用户看到二维坐标空间和基本工具，而不是完整管理界面。
- 函数、几何、参数、列表、手绘、区域、样式应被视为对象能力体系的一部分，而不是彼此孤立的子系统。
- Apple Pencil 默认是背景绘画工具，不自动转函数，不默认参与数学计算。
- 手绘数学化必须由用户主动触发，不能把用户的背景线条误当成函数识别结果。
- 数学对象可以选择艺术笔触，但数学定义本身必须保持精确。
- 样式不是附属装饰，而是 Plane 数学艺术表达层的一部分。
- 参数动画应进入 v1.0 的基础范围，但只限参数播放和基础动态表现；完整时间轴、关键帧和多轨道系统应后置。
- 填色区域允许智能封口，但不能破坏原始可见边界，也不能把虚拟封口误写成真实几何边。
- 坐标系保持标准、清晰、数学化，不采用手绘坐标系或艺术坐标系。

## 4. v1.0 Must Have

以下能力定义的是 Plane v1.0 的主范围目标。若现实实现尚不稳定，应理解为 `v1.0 target`，而不是“当前已经完整实现”或“当前代码已经完整闭环”。

| 主题 | 能力 | v1.0 要求 | 现实备注 |
|---|---|---|---|
| 默认界面 | 标准二维坐标画布 | Must Have | 作为默认打开界面的中心区域 |
| 默认界面 | 基础工具栏 | Must Have | 至少覆盖选择、平移、点、线、圆、函数、参数等基础入口 |
| 默认界面 | 函数 / 表达式输入入口 | Must Have | 入口应直接可见，不应藏得过深 |
| 默认界面 | 对象 / 表达式面板默认收起或轻量显示 | Must Have | 不能默认铺满复杂列表 |
| 默认界面 | Inspector 在选中对象后出现 | Must Have | 默认不抢占主画布 |
| 默认界面 | 样式面板按需展开 | Must Have | 不要求默认常驻展开 |
| 基础对象 | `point` | Must Have | 已有真实对象基础，可作为 v1.0 核心对象 |
| 基础对象 | `segment` | Must Have | 已有真实对象基础，可直接纳入主范围 |
| 基础对象 | `line` | Must Have | 已有真实对象基础，可直接纳入主范围 |
| 基础对象 | `ray` | Must Have | 已有真实对象基础，可直接纳入主范围 |
| 基础对象 | `circle` | Must Have | 已有真实对象基础，可直接纳入主范围 |
| 基础对象 | `arc` | Must Have | 已有真实对象基础，可直接纳入主范围 |
| 基础对象 | `function` | Must Have | 已有真实对象基础，应继续作为主入口能力 |
| 基础对象 | `parameter / slider` | Must Have | `parameterGroup` 不计入本主范围 |
| 函数与曲线 | 显函数 | Must Have | Plane 当前已有真实基础，可作为稳定主入口 |
| 函数与曲线 | 点列 / 折线 | Must Have | 作为函数表达与离散曲线能力的一部分 |
| 函数与曲线 | 参数曲线 | Must Have | 高优先级 `v1.0 target`，属于主范围目标，不写成“已完整完成” |
| 函数与曲线 | 极坐标曲线 | Must Have | 高优先级 `v1.0 target`，保留在 Must，而不是写成当前已完成 |
| 参数 | `slider` | Must Have | 参数入口必须直观存在 |
| 参数 | 参数手动调节 | Must Have | 用户可直接拖动或调整参数 |
| 参数 | 参数播放 | Must Have | 作为动画底座的一部分 |
| 参数 | 播放 / 暂停 | Must Have | 最低动画控制闭环 |
| 参数 | 循环 / 往返 | Must Have | 基础播放模式 |
| 参数 | 速度调节 | Must Have | 至少提供基础速度控制 |
| 样式 | 颜色 | Must Have | 基础样式闭环 |
| 样式 | 透明度 | Must Have | 数学艺术表达必须保留 |
| 样式 | 线宽 | Must Have | 基础样式闭环 |
| 样式 | 线型 | Must Have | 基础样式闭环 |
| 样式 | 点样式 | Must Have | 点对象需要最基本样式控制 |
| 样式 | 填充色 | Must Have | 区域填色基础 |
| 样式 | 填充透明度 | Must Have | 区域表达基础 |
| 保存与导出 | 项目保存 / 打开 | Must Have | 必须形成文档闭环 |
| 保存与导出 | 缩略图 | Must Have | 用于项目管理与预览 |
| 保存与导出 | PNG | Must Have | 第一阶段核心导出格式 |
| 保存与导出 | 透明背景 PNG | Must Have | 数学艺术创作的重要导出能力 |
| 保存与导出 | 高分辨率导出 | Must Have | 必须进入 v1.0 导出闭环 |

## 5. v1.0 Should Have

以下能力建议进入 v1.0，但若工程风险、稳定性或联动复杂度过高，可以拆成分批落地，放入 v1.0 后段或 v1.0x 小版本；本节不应被偷渡解释为 Must。

| 主题 | 能力 | 说明 |
|---|---|---|
| 面板 | 轻量对象与表达式面板 | 统一整理函数、参数、几何对象和区域对象 |
| 面板 | 自动分类：函数、参数、几何对象、列表与表格、手绘与拟合、区域与填充 | 默认分类要服务整理，而不是堆叠复杂 UI |
| 面板 | 用户自定义分组 / 文件夹 | 支持用户按创作逻辑组织对象 |
| 列表与表格 | 基础列表：数值列表、点列表 | 作为后续批量控制的第一步 |
| 列表与表格 | 基础表格：点表、数据表 | 先做基础承载，不追求完整数据系统 |
| 曲线 | 基础曲线族 | 支持基础批量曲线表达 |
| Pencil | Apple Pencil 背景绘画层 | 用于草图、标注、背景纹理、构图辅助 |
| Pencil | 简单画笔、橡皮、颜色、透明度、粗细 | 仅覆盖背景绘画的最小工具集 |
| Pencil | 背景绘画层锁定 / 隐藏 / 是否导出 | 保障背景层可控，不干扰数学对象 |
| 样式 | 函数笔触：标准线、手绘线、点线等基础笔触 | 强化艺术表达，但不改变数学定义 |
| 区域 | 基础区域填色 | 让区域能力进入可用阶段 |
| 区域 | 多曲线组合区域 | 多条曲线可共同构成填色区域 |
| 区域 | 两曲线之间区域 | 兼具数学意义与构图价值 |
| 区域 | 智能封口：端点接近时建立不可见填色边界 | 只用于填色判断，不破坏原边界 |
| 样式 | 基础颜色渐变 | 作为数学艺术表达增强层 |
| 样式 | 基础透明度渐变 | 不只是装饰，应保留为正式能力 |
| 动态 | 轨迹残影 | 与参数播放协同，形成动态表达闭环 |

## 6. v1.0 Foundation / Reserved Only

以下内容需要在概念、数据模型、文档结构或保存语义上预留，但不要求 v1.0 完整 UI、完整交互或完整功能闭环。

| 预留项 | 预留要求 |
|---|---|
| 动画数据模型预留 | 为参数播放、对象动态变化和后续高级动画保留结构空间 |
| 样式渐变驱动方式预留：参数、列表编号、距离、曲线方向、时间 | 预留渐变驱动语义和保存表达，不要求 v1.0 全部做满 |
| 手绘拟合结果对象预留 | 为平滑路径、点列、拟合曲线等后续输出对象预留对象语义 |
| 列表控制函数取值范围预留 | 预留“列表控制显示区间”的语义和存储表达 |
| 列表控制曲线参数范围预留 | 预留“列表控制参数区间”的语义和存储表达 |
| 区域对象依赖边界预留 | 预留区域引用边界对象的结构，不要求完整编辑器 |
| 虚拟封口边界作为区域内部边界预留 | 只作为区域内部语义，不当作普通几何对象暴露 |
| SVG / GIF / MP4 / 序列帧导出预留 | 预留未来导出格式的扩展位，不等于 v1.0 支持这些格式 |
| 傅里叶 / 级数拟合预留 | 作为 eMathica 特色方向保留概念入口和输出对象空间 |
| 复杂约束系统预留 | 不在 v1.0 实现完整求解器，但不阻断未来扩展 |

## 7. v1.x Follow-up

以下能力不进入 v1.0 范围冻结，但属于 v1.x 的自然延伸，应该在 v1.0 范围文档中被明确保留为后续用户可见能力，而不是仅停留在“结构预留”层。

- 列表控制函数显示区间
- 列表控制参数曲线参数区间
- 列表生成区域族
- 更完整的曲线族
- 手绘转路径 / 点列 / 参数曲线
- 样条拟合
- 多项式拟合
- 傅里叶 / 级数拟合作为特色功能
- 点击区域自动识别边界
- 显示虚拟封口预览
- 手动编辑虚拟封口
- SVG 导出作为正式静态矢量格式
- GIF 导出作为基础动态导出
- 更完整的透明度渐变和颜色渐变
- 更强的样式表达式

## 8. Later / Not for v1.0

以下能力属于更长期方向，不应被塞入 v1.0 目标，也不应在本轮范围文档中伪装成“第一版应完成”或“仅靠范围预留即可覆盖”。

- 完整时间轴
- 多轨道动画
- 关键帧曲线编辑
- MP4 导出
- 序列帧导出
- 复杂自交区域填色
- 多孔区域
- 区域布尔运算
- 完整专业绘画系统
- 高级笔刷系统
- 完整几何识别系统
- 完整约束求解器
- 复杂依赖图可视化
- 示例作品集

关于“示例作品集”的特别说明：

- 它更适合在 Notes Calculator 有基础图文、公式说明和排版能力之后再制作。
- Plane v1.0 可以保留少量轻量模板，但不应把完整作品集制作纳入本次主范围。

## 9. Explicitly Out of Scope

- 不做手绘坐标系或艺术坐标系。
- 不让背景笔迹自动转函数。
- 不一打开就展示大量表达式、图层和高级面板。
- 不把 Plane 写成独立输入系统；输入系统应使用 `MathInputKit`。
- 不把 `Document / Workspace / Animation / Style` 的共享能力写成 Plane 私有能力。
- 不实现完整视频编辑器。
- 不追求一次性复刻 Desmos 的全部能力。
- 不把 `Data / Music / Notes / Modeling` 纳入 Plane v1.0 实施范围。

## 10. Relationship to Shared Systems

以下能力属于共享系统。本节的目的，是防止在 Plane v1.0 scope 中把共享层能力误写成 Plane 私有子系统：

| 能力 | 归属 |
|---|---|
| 结构化输入 | `MathInputKit` |
| 表达式模型 | `MathCore` |
| 保存 / 打开 | `DocumentKit` |
| 选择 / Inspector / 工具系统 | `WorkspaceKit` |
| 样式 tokens | `ThemeKit / Object Style` |
| 参数 / 动画模型 | `Shared Calculator / Animation foundation` |
| 导出 | `Shared Export layer` |
| 对象模型 | `MathCore / ObjectSystem` |

Plane 的职责应集中在二维实现层，而不是重新实现这些共享能力：

- 在二维坐标系中表现这些对象。
- 提供二维渲染。
- 提供二维命中。
- 提供二维几何构造。
- 提供二维区域填色和智能封口。
- 提供二维动态表现。

## 11. Relationship to Current Reality Map

本文件沿用 `R1 / R2 / R3` 的 Reality Map，不重新做架构审计，也不推翻已有现实结论。范围判断应以“现实边界 + 第一版目标”共同决定，而不是把未来设计直接写成当前已交付。

当前现实对照应理解为：

- Plane 是当前最成熟的 Calculator，已经具备一批真实 2D 对象与服务基础。
- Plane 当前已有真实对象包括：`point`、`function`、`circle`、`segment`、`line`、`ray`、`arc`、`parameter`。
- `parameterGroup` 仍是 stub，不进入 v1.0 主范围。
- Plane 当前已有命中、渲染、预览、采样、语义以及部分依赖服务，这解释了为什么 Plane 可以承担 v1.0 主线，但不等于所有上层系统都已完整闭环。
- Space 当前只应视为 MVP 骨架，不作为本次范围主线。
- `Data / Music / Notes / Modeling` 当前仍是 placeholder-only，不进入本次 Plane v1.0 范围。
- 全局 `DependencyGraph` 不是完整实现，因此本文件不能把全局依赖系统写成“已完成能力”。
- 动画系统当前不是完整实现，因此本文件只能把“参数播放与动画底座”写成 v1.0 目标与预留方向，不能写成当前已完整落地。
- `MathInputKit`、`WorkspaceKit`、`DocumentKit` 都是真实存在的 Package；输入、Workspace、Document 不应被改写成 Plane 私有能力。
- `DocumentSystem` shadow copy 问题不在本文件展开，但后续如处理保存链路，应注意 `R2` 对 `DocumentKit` 与本地 shadow copy 的现实提醒。
- `CapabilityRegistry` 不在本任务中修改，本文件也不承担 Registry 校正职责。

## 12. Acceptance Criteria for v1.0 Scope Freeze

当本文件可用于范围冻结时，应满足以下条件：

- 可以据此继续拆出 `Plane_v1.0_CodexTasks.md`。
- 每个主要功能都已有 `Must Have / Should Have / Reserved Only / v1.x Follow-up / Later / Out of Scope` 分类。
- 你此前明确指定到 `Must` / `Should` 的条目仍保留在对应章节，没有被悄悄重排。
- 没有把完整 `PlaneDesign.md` 的功能池直接塞进 v1.0。
- 没有把未来能力写成当前已实现。
- `极坐标曲线` 仍在 `Must Have`，并且被明确写成高优先级 `v1.0 target`。
- `Reserved Only` 描述的是现在必须预留的语义、结构或保存表达，而不是已经承诺交付的完整用户功能。
- `v1.x Follow-up` 描述的是后续用户可见能力，而不是和 `Reserved` 重复的结构占位。
- 没有要求 Codex 在本轮修改 Swift 源码、Package、`CapabilityRegistry` 或项目配置。
- 没有与 `R1 / R2 / R3` 的 Reality Map 发生正面冲突。
